# 에러 처리

## 에러 타입

| 에러 | 패키지 | throw 시점 |
|-------|---------|------------|
| `ValidationError` | `@routar/core` | 요청 또는 응답 검증 실패 |
| `HttpError` | `@routar/core` | 서버가 비-2xx 상태 반환 (모든 executor — fetch, Axios, ky) |
| `TimeoutError` | `@routar/core` | `createFetchExecutor`의 `timeout` 초과 |

모든 트랜스포트는 HTTP 실패를 `HttpError`로 정규화합니다. Axios와 ky executor는 각자의 네이티브 에러(`AxiosError`, ky `HTTPError`)를 잡아 응답에서 `status`·`statusText`·`body`를 채운 `HttpError`로 다시 throw합니다. 원본 트랜스포트 에러는 `err.cause`에 보존되므로 필요할 때 `AxiosError` 고유 필드에 접근할 수 있습니다.

## 에러 처리

```ts
import { TimeoutError, ValidationError, HttpError } from '@routar/core'

try {
  await todoApi.create({ body: { title: '' } })
} catch (err) {
  if (err instanceof ValidationError) {
    console.log(err.message, err.cause)
  }
  if (err instanceof HttpError) {
    console.log(err.status)     // e.g. 422
    console.log(err.statusText) // e.g. 'Unprocessable Entity'
    console.log(err.body)       // 파싱된 JSON 페이로드, 또는 null
    console.log(err.url)        // 전체 요청 URL, e.g. 'https://api.example.com/todos'
    console.log(err.method)     // HTTP 메서드, e.g. 'POST'
    console.log(err.cause)      // 원본 트랜스포트 에러 (AxiosError, ky HTTPError), 또는 undefined
  }
  if (err instanceof TimeoutError) {
    console.log(err.ms)
  }
}
```

## 트랜스포트에 무관한 HTTP 에러

모든 executor가 HTTP 실패를 `HttpError`로 정규화하므로, `instanceof HttpError` 한 번으로 fetch·Axios·ky를 모두 처리할 수 있습니다. `isAxiosError`나 ky의 `HTTPError`로 분기할 필요가 없습니다:

```ts
import { HttpError } from '@routar/core'

try {
  await todoApi.getDetail({ path: { id: 1 } })
} catch (err) {
  if (err instanceof HttpError) {
    // 어떤 executor든 — 비-2xx 응답
    console.log(err.status, err.body)
  }
}
```

트랜스포트 고유 필드(예: `AxiosError`의 `config`나 `code`)가 필요하면 `err.cause`로 접근하세요:

```ts
import { HttpError } from '@routar/core'
import { isAxiosError } from 'axios'

try {
  await todoApi.getDetail({ path: { id: 1 } })
} catch (err) {
  if (err instanceof HttpError && isAxiosError(err.cause)) {
    console.log(err.cause.code)   // 예: 'ERR_BAD_REQUEST'
    console.log(err.cause.config) // 원본 요청 설정
  }
}
```

`onError` 플러그인도 동일하게 정규화된 `HttpError`를 받으므로 플러그인 코드 역시 트랜스포트에 무관합니다. (응답이 없는 에러 — 네트워크 실패나 취소 — 는 정규화할 HTTP 상태가 없으므로 원본 그대로 다시 throw됩니다.)

## 서버 에러만 재시도

`retry`는 `createFetchExecutor`의 옵션입니다:

```ts
createFetchExecutor('https://api.example.com', {
  retry: {
    count: 3,
    shouldRetry: (err) => {
      if (err instanceof HttpError) return err.status >= 500
      return true
    },
  },
})
```
