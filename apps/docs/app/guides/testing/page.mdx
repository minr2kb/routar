# Testing

Because routar separates the endpoint spec from the HTTP transport, tests can inject a mock executor and skip the network entirely — no server, no MSW setup, no flaky latency.

## Mock executor

A mock executor is a plain `createExecutor` call whose transport function returns test data directly.

```ts
import { createApi, createExecutor } from '@routar/core'
import { TodoRouter } from '@/remote/services/todo/todo.api'

const mockExecutor = createExecutor(async ({ url, method }) => {
  if (url.endsWith('/todos') && method === 'GET') {
    return [{ id: 1, title: 'buy milk', completed: false }]
  }
  if (url.match(/\/todos\/\d+$/) && method === 'GET') {
    return { id: 1, title: 'buy milk', completed: false }
  }
  throw new Error(`Unhandled: ${method} ${url}`)
})

const todoApi = createApi(mockExecutor, TodoRouter)
```

`createApi` with the mock executor gives you the same typed API surface — `todoApi.getList({})`, `todoApi.getDetail({ path: { id: 1 } })` — with Zod validation still running on every response.

```ts
const todos = await todoApi.getList({})
// → [{ id: 1, title: 'buy milk', completed: false }]
//   validated by TodoListSchema at runtime
```

If the mock returns the wrong shape, Zod throws a `ValidationError` immediately — the same as in production. Your tests catch bad mock data before it reaches the component.

## Per-test executor swap

Create the API inside each test to isolate behavior:

```ts
import { createApi, createExecutor } from '@routar/core'
import { ValidationError } from '@routar/core'

function makeTodoApi(data: unknown) {
  return createApi(
    createExecutor(async () => data),
    TodoRouter,
  )
}

test('renders todo list', async () => {
  const api = makeTodoApi([{ id: 1, title: 'buy milk', completed: false }])
  const todos = await api.getList({})
  expect(todos).toHaveLength(1)
})

test('throws ValidationError on bad shape', async () => {
  const api = makeTodoApi([{ wrong: true }])
  await expect(api.getList({})).rejects.toBeInstanceOf(ValidationError)
})
```

## Error scenario testing

Return a rejected promise to simulate network failures or server errors:

```ts
const errorExecutor = createExecutor(async () => {
  throw new Error('Network error')
})

test('handles fetch failure', async () => {
  const api = createApi(errorExecutor, TodoRouter)
  await expect(api.getList({})).rejects.toThrow('Network error')
})
```

## Mock executor vs MSW

Both approaches work — they intercept at different layers:

| | Mock executor | MSW |
|---|---|---|
| **Intercepts at** | routar transport layer | Network (service worker / Node interceptor) |
| **HTTP actually sent** | No | Yes |
| **Tests URL construction** | No | Yes — real URL must match handler |
| **Tests headers** | No | Yes — real headers visible to handler |
| **Setup cost** | Minimal | MSW server + handler registration |
| **Best for** | Component / hook unit tests | Integration tests, contract verification |

Use a mock executor when the goal is to test how your component or hook handles API responses. Use MSW when you want to verify that the right URL, method, or headers are sent — or when testing the full stack from fetch to UI.

## With dispatchExecutor

If you use `dispatchExecutor` in your production executor, you can extend it to pick a mock executor in test environments without changing any call sites:

```ts
// executor.ts
export const apiExecutor = dispatchExecutor(() => {
  if (process.env.NODE_ENV === 'test') return mockExecutor
  return typeof window === 'undefined' ? serverExecutor : clientExecutor
})
```

The API client (`createApi(apiExecutor, router)`) stays the same in production, tests, SSR, and CSR — only the transport changes.
